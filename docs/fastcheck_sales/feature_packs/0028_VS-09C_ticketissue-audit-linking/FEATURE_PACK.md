## 1. Purpose

VS-09C links the attendee rows created/reused by VS-09B to durable Sales-side ticket audit records:

```text
verified paid Sales order
  -> FastCheck.Tickets.Issuer
  -> existing Ecto Attendee rows from VS-09B
  -> FastCheck.Sales.TicketIssue rows
```

This is the first slice where `FastCheck.Sales.TicketIssue` becomes meaningful.

The key rule:

```text
Attendee is scanner/runtime truth.
TicketIssue is Sales audit/link/token/validity truth.
DeliveryAttempt is delivery history truth.
```

Do not collapse these responsibilities.

## 8. Issuer Workflow

Preferred flow:

```text
FastCheck.Tickets.Issuer.issue_order(order_id, opts)
  -> lock/load Sales.Order
  -> verify paid_verified or fulfillment_queued state
  -> load OrderLine rows
  -> call/reuse VS-09B attendee bridge result
  -> for each issuance unit:
       find existing TicketIssue by sales_order_line_id + line_item_sequence
       if exists, verify attendee_id and ticket_code match expected unit
       if missing, create TicketIssue with attendee_id, ticket_code, token hashes
  -> if all expected issues exist:
       mark Order ticket_issued according to legal transition matrix
       append StateTransition records
       return {:ok, result}
  -> if partial mismatch/failure:
       mark partially_issued or manual_review according to VS-09A
       return recoverable error with correlation_id
```

Rules:

```text
TicketIssue creation must be idempotent forever.
A duplicate worker must not create duplicate TicketIssue rows.
A retry after crash must reuse existing TicketIssue rows.
If an existing TicketIssue points to a different Attendee than the deterministic unit expects, move to manual_review and stop.
If Attendee is not active/scanner-compatible after VS-09B, move to manual_review and do not issue.
```


```text

```

Concurrency protection:

```text
order/advisory lock + DB unique constraints
Oban uniqueness helps scheduling but is not correctness
TicketIssue uniqueness must enforce `sales_order_line_id + line_item_sequence`
Attendee uniqueness remains event_id + ticket_code plus VS-02/VS-09B Sales-origin key
```

## 10. Order State Rules

Allowed outcomes:

```text
All issuance units linked -> Order.mark_ticket_issued
Some units linked, some failed -> Order.mark_partially_issued or mark_manual_review per VS-09A
No units linked due invalid precondition -> no issue rows, return error
Existing full issue set on retry -> idempotent success
```

Preconditions:

```text
Order status must be paid_verified or fulfillment_queued according to VS-00A/VS-09A.
PaymentAttempt must already be verified_success from VS-07B/VS-07C.
Inventory must already be consumed/recovered before issuance; do not mutate Redis here.
Attendee bridge must return deterministic attendee IDs for every expected unit.
```

Forbidden transitions:

```text
draft -> ticket_issued
awaiting_payment -> ticket_issued
payment_pending -> ticket_issued
manual_review -> ticket_issued without explicit recovery action/reason
expired/cancelled/refunded -> ticket_issued
```

## 16. Failure Modes

| Failure | Required behavior |
|---|---|
| Duplicate worker starts | Reuse existing Attendees and TicketIssues; return idempotent success. |
| Attendee exists but TicketIssue missing | Create only the missing TicketIssue. |
| TicketIssue exists but attendee_id mismatch | Manual review; do not overwrite. |
| Token hash generation fails | Do not mark ticket issued; retry or manual review. |
| DB unique conflict on TicketIssue | Reload existing row and verify deterministic ownership. |
| Some issue rows created then transaction fails | Retry must complete missing rows or reuse existing rows. |
| Order not verified | No TicketIssue rows. |
| Scanner/mobile tests fail | Stop; this slice must not change scanner behavior. |
